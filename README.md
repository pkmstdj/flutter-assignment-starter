# 국내 주식 관심종목 앱 — 구현물

㈜이든크루 Flutter 개발자 과제 구현 결과입니다.

## 실행 & 검증

```bash
flutter run                 # 앱 실행
flutter analyze             # 정적 분석
flutter test                # 전체 테스트
flutter run -t test/demo/main_demo.dart   # mock 데이터 데모
```

- `flutter analyze` → No issues
- `flutter test` → 골든 6개(화면 픽셀 비교)를 제외한 기능 테스트 전부 통과
  - 골든 실패는 코드 결함이 아니라 OS 간 폰트 렌더링(안티에일리어싱) 차이입니다. 근거: (1) 손대지 않은 빈 화면 골든도 실패, (2) diff가 글자 테두리에만 몰림.

## 구현 범위

과제의 3영역을 모두 구현했습니다.

1. **데이터 연동** — 네이버 검색/실시간/메타/일별시세(HTML) 연동, DTO 파싱, repository 변환·조합
2. **UI 구현** — 검색 결과 행(하이라이트/하트/액션바), 검색 토스트(glass blur), 날짜 선택 바텀시트(휠 피커)
3. **상태 동기화** — 즐겨찾기 ↔ 검색 결과 하트, 날짜 변경 시 목록·상세 동시 갱신

## 주요 고려사항 (요약)

- **레이어 분리 + 의존성 역전**: 네이버 응답의 불안정성(문자열 숫자, EUC-KR HTML, 수백 페이지 시세)을 data 레이어 안에 가두고, presentation은 안정적인 도메인 모델·인터페이스만 보게 설계했습니다. → fake 주입으로 네트워크 없이 결정론적 테스트가 가능합니다.
- **실사용 가능한 거래일 로딩**: 실제 종목은 lastPage가 수백(예: 삼성전자 751)이라 전량 로드 시 앱이 멈춥니다. 최근 구간만 상한(파라미터화)으로 받고, 기본 목록은 거래일 축이 필요 없는 빠른 경로로 분리했습니다.
- **graceful degradation**: 실시간 시세 실패 시 과거 종가로 폴백해 화면이 죽지 않게 처리했습니다.
- **상태 단일 원천(single source of truth)**: 선택 날짜를 한 곳에 두고 목록·상세를 함께 갱신해 불일치를 원천 차단했습니다.

각 구현의 트레이드오프와 근거는 코드 내 한국어 주석으로 남겨 두었고, 면접에서 원하시는 부분을 짚어 주시면 상세히 설명드리겠습니다.

## AI 도구 활용 방식

안내 주신 대로, AI 도구 활용 방식과 이해도를 함께 말씀드립니다.

- **활용 범위**: AI는 **초안 생성·사고 정리(러버덕)** 용도로 사용했습니다. 정규식 초안, 아키텍처 후보 비교, 테스트 케이스 목록 등에서 도움을 받았고, **이 문서의 내용 정리·문장 다듬기에도 AI를 활용**했습니다.
- **직접 판단한 부분**: 아키텍처 경계, 캐시 전략, 엣지케이스 대응, 버그 원인 특정, 최종 검증은 직접 수행했습니다. AI가 준 코드도 **테스트로 반증하거나 근거를 확인한 뒤에만** 남겼습니다.
- **한 가지 예**: 처음엔 스펙 문구대로 "lastPage 끝까지 로드"하도록 되어 있었으나, 실제 실행에서 앱이 멈추는 것을 확인하고 "스펙의 의도는 거래일 축 확보"라고 재해석해 페이지 상한 + 빠른 경로 분리로 바꿨습니다.
- **이해도**: 제출한 코드는 어느 부분이든 **왜 그렇게 구현했는지 / 어디서 깨질 수 있는지 / 대안은 무엇이었는지** 라인 단위로 설명 가능합니다. 필요 시 면접에서 원하시는 함수를 짚어 주시면 바로 설명드리겠습니다.
